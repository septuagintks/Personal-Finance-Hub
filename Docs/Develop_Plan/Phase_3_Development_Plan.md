# Personal Finance Hub (PFH) - Phase 3 Development Plan

Version: 1.0
Backend: C++23
Architecture: Clean Architecture + Lightweight DDD
Status: Draft

---

## 1. 导言与阶段目标

Phase 3 是预留扩展阶段，用于在核心记账能力和产品体验稳定后，接入外部账单、支付平台、银行数据和自动化同步能力。

### 1.1 阶段原则

- **只读优先**：外部平台接入优先读取账单和交易数据，不做支付、转账等写操作。
- **用户确认优先**：外部数据进入正式账本前，必须经过预览、去重、校验和用户确认。
- **幂等与可追踪优先**：所有导入和同步任务必须具备幂等指纹、任务状态和审计记录。
- **核心能力不被外部依赖绑架**：平台接入失败不得影响本地记账、报表和历史数据访问。

### 1.2 阶段范围

Phase 3 包含：

- CSV、Excel、PDF、邮件账单等账单导入能力。
- 支付平台、银行、信用卡或第三方聚合平台的只读数据接入。
- Provider 插件化同步框架。
- 幂等指纹、冲突处理、预览确认和导入审计。
- 自动分类规则、商户标准化和批量修正。
- 异常交易提醒、订阅识别、预算提醒和现金流预测等高级自动化能力。

Phase 3 不包含：

- 真实支付、转账、扣款等资金写操作。
- 绕过用户授权的账户抓取。
- 不可追踪的自动入账。
- 未经合规评估的敏感数据长期保存。

## 2. 架构定位与职责边界

### 2.1 Phase 3 分层路径

```text
External Provider / File Import
  -> Sync Adapter / Import Parser
    -> Sync Application Use Case
      -> Domain Validation / Deduplication
      <- Repository / Import Staging / Audit
        -> PostgreSQL
```

### 2.2 职责边界

- Provider Adapter 负责外部协议、文件格式和平台字段映射，不直接写正式账本。
- Import Parser 负责解析原始账单，输出标准化候选流水。
- Application 层负责导入任务、预览确认、冲突处理和事务边界。
- Domain 层负责金额、币种、分类、账户归属和重复检测规则。
- Infrastructure 层负责任务状态、原始记录、暂存表、审计和重试。

## 3. 里程碑计划

### 3.1 P3.0 外部接入边界与安全基线

目标：

- 明确 Phase 3 的外部数据接入边界、授权方式和合规要求。
- 确认只读优先策略和用户确认流程。
- 设计导入暂存区和审计链路。

交付物：

- 外部接入安全基线。
- 导入暂存模型。
- 用户授权与撤销策略。

验收标准：

- 未授权数据不得进入系统。
- 外部平台失败不影响核心记账。
- 敏感凭据不得明文存储。

### 3.2 P3.1 文件账单导入基础

目标：

- 支持 CSV 和 Excel 账单导入。
- 提供字段映射、数据预览、格式校验和用户确认。
- 建立导入任务状态机。

交付物：

- CSV 导入器。
- Excel 导入器。
- 字段映射配置。
- 导入预览与确认流程。

验收标准：

- 金额必须通过字符串进入 `Decimal`。
- 导入前可预览候选流水。
- 用户确认前不写入正式账本。
- 导入失败可定位到具体行和字段。

### 3.3 P3.2 幂等、去重与冲突处理

目标：

- 建立外部流水幂等指纹。
- 处理重复导入、跨账户冲突、时间误差和商户名差异。
- 支持用户手动合并、跳过和修正。

交付物：

- `ImportFingerprint` 或等价幂等组件。
- 冲突检测规则。
- 导入冲突处理流程。
- 导入审计日志。

验收标准：

- 重复导入同一账单不会产生重复流水。
- 幂等键必须包含 `user_id`，必要时包含账户或外部账户标识。
- 冲突处理结果可追踪。

### 3.4 P3.3 支付平台与银行只读接入

目标：

- 建立 Provider 插件化接口。
- 优先接入一个低风险只读数据源作为样板。
- 支持授权、拉取、分页、限流、失败重试和撤销授权。

交付物：

- Provider Adapter 接口。
- 示例 Provider。
- 授权与撤销流程。
- 同步任务状态管理。

验收标准：

- Provider 失败不会影响其他 Provider 或核心记账。
- 同步任务可重试、可取消、可审计。
- 外部原始数据和标准化候选流水可关联追踪。

### 3.5 P3.4 自动分类与规则引擎

目标：

- 根据商户、关键词、账户、金额区间和历史行为提供分类建议。
- 支持用户维护规则并批量应用。
- 保留自动建议和用户最终确认的差异。

交付物：

- 规则引擎基础。
- 商户标准化表。
- 分类建议服务。
- 批量修正流程。

验收标准：

- 自动分类只给建议，不绕过用户确认直接修改高风险数据。
- 用户规则优先级明确。
- 批量修改可审计、可回滚或可补偿。

### 3.6 P3.5 高级自动化与提醒

目标：

- 增加订阅识别、异常交易提醒、预算提醒和现金流预测。
- 将提醒设计为可配置、可关闭、可追踪。
- 避免把预测结果当作确定财务事实。

交付物：

- 订阅识别规则。
- 异常交易检测规则。
- 预算提醒任务。
- 现金流预测原型。

验收标准：

- 提醒不修改原始财务事实。
- 用户可以关闭或调整提醒规则。
- 预测结果必须带有时间范围和不确定性说明。

### 3.7 P3.6 扩展能力稳定化

目标：

- 补齐 Provider 测试夹具、导入回归样例和同步监控。
- 建立外部接入故障隔离和数据修复流程。
- 准备后续新增 Provider 的开发指南。

交付物：

- Provider 开发指南。
- 导入样例数据集。
- 同步任务监控。
- 外部接入故障处理手册。

验收标准：

- 新增 Provider 不需要修改核心领域模型。
- 外部接入失败具备降级和恢复策略。
- 导入与同步行为可通过测试复现。

## 4. 质量门禁

### 4.1 必跑检查

Phase 3 合并前至少执行：

```text
backend build
unit tests
import parser tests
provider contract tests
repository integration tests
api tests
e2e import smoke tests
markdown check
```

### 4.2 阻断规则

- 未经用户确认直接写入正式账本，禁止合并。
- 幂等键缺少 `user_id`，禁止合并。
- 外部凭据明文存储，禁止合并。
- Provider 失败影响核心记账路径，禁止合并。
- 导入金额经由 `float` 或 `double`，禁止合并。

## 5. 风险与应对

### 5.1 外部平台不稳定

风险：平台接口、导出格式、限流策略和授权方式可能频繁变化。

应对：

- Provider 插件化隔离变化。
- 保留原始导入记录和标准化记录。
- 优先实现文件导入，再推进在线同步。

### 5.2 重复导入和误匹配

风险：银行账单和支付平台账单可能字段不完整，导致重复流水或误合并。

应对：

- 使用包含 `user_id`、账户、时间、金额、商户和来源的复合指纹。
- 导入前提供预览。
- 冲突处理必须由用户确认。

### 5.3 合规与隐私风险

风险：外部账单、支付记录和银行数据包含高度敏感信息。

应对：

- 最小化保存外部敏感字段。
- 凭据加密存储，并支持撤销授权。
- 审计外部数据访问和导入行为。

## 6. 待决策选项

当前没有必须由维护者立即选择的架构项。

### 6.1 后续可评估事项

- **首个 Provider 选择**：建议先选 CSV/Excel 文件导入作为最低风险样板，再评估支付平台或银行接入。
- **PDF/OCR 能力边界**：PDF 和 OCR 解析不稳定，建议等文件导入链路稳定后再推进。
- **自动分类策略**：先规则引擎，后续再评估模型推荐，避免早期引入不可解释分类结果。
