# Personal Finance Hub (PFH) - Phase 2 Development Plan

Version: 1.0
Backend: C++23
Frontend: Vue 3
Architecture: Clean Architecture + Lightweight DDD
Status: Draft

---

## 1. 导言与阶段目标

Phase 2 的目标是在 Phase 1 后端最小闭环稳定之后，把 PFH 从“可正确记账的后端基础版本”推进到“可日常使用、可观察、可维护的产品版本”。

### 1.1 阶段原则

- **体验建立在正确性之上**：不重写 Phase 1 的金融规则，只围绕稳定模型增强交互、报表和维护能力。
- **前后端协同演进**：前端页面、DTO、API 错误格式和用户偏好必须同步推进。
- **维护动作可审计**：归档、危险删除、余额重建、数据修正等操作必须留下审计线索。
- **稳定性优先于功能堆叠**：后台任务、错误追踪、日志、告警和测试门禁必须进入阶段交付。

### 1.2 阶段范围

Phase 2 包含：

- Vue 3 前端应用骨架与核心页面。
- 账户、流水、转账、分类、标签和用户偏好管理体验。
- 净资产、现金流、分类统计和 Dashboard Summary 报表增强。
- 账户归档、危险删除确认、余额缓存重建和历史汇率补全。
- 审计日志、后台任务监控、全局错误边界和运维可观测性。
- 端到端 smoke test、覆盖率门禁、性能基准和文档 lint。

Phase 2 不包含：

- 银行、信用卡、支付平台的完整自动同步。
- OCR、PDF、邮件账单解析。
- 真实支付平台写操作。
- 复杂机器学习自动分类。

## 2. 架构定位与职责边界

### 2.1 Phase 2 分层路径

```text
Frontend Vue 3
  -> REST API / DTO
    -> Application Use Case / QueryService
      -> Domain Rules
      <- Infrastructure Repository / Job / Audit
        -> PostgreSQL
```

### 2.2 职责边界

- Frontend 负责交互、状态管理、展示格式和用户操作确认，不承载金融计算规则。
- Presentation 层保持统一 API 响应、认证上下文和错误映射。
- Application 层继续作为业务编排入口，负责权限、事务、审计和任务触发。
- Infrastructure 层负责后台任务、审计落库、缓存重建和查询优化。
- Domain 层规则不得为了前端展示便利而被削弱或绕过。

## 3. 里程碑计划

### 3.1 P2.0 阶段基线与接口复核

目标：

- 复核 Phase 1 API、DTO 和错误格式是否满足前端产品化需要。
- 明确 Phase 2 页面结构、导航、状态管理和接口依赖。
- 补充需要从 Phase 1 延伸出来的 QueryService 和维护接口。

交付物：

- Phase 2 页面与 API 对照清单。
- 前端路由、Store 和组件目录草案。
- API 差距清单。

验收标准：

- 所有 Phase 2 页面都能映射到已有或计划新增的 API。
- 金额、日期、币种和错误响应格式在前后端之间保持一致。
- 不引入新的金融计算规则分叉。

### 3.2 P2.1 前端应用骨架

目标：

- 创建 Vue 3 + Vite 前端骨架。
- 接入路由、状态管理、HTTP Client、认证上下文和基础布局。
- 建立前端本地配置、环境变量和错误处理入口。

交付物：

- 前端项目结构。
- 登录态管理与 API Client。
- 基础页面框架和导航。

验收标准：

- 用户可以登录、刷新 Token、退出登录。
- 401 响应能触发统一登录态处理。
- 前端金额输入组件只提交字符串金额。

### 3.3 P2.2 核心记账体验

目标：

- 完成账户列表、账户详情、流水列表、流水录入和转账表单。
- 支持分类、标签、过滤、搜索和基础批量操作。
- 将后端业务错误清晰反馈给用户。

交付物：

- 账户管理页面。
- 流水录入与编辑页面。
- 转账表单。
- 分类与标签基础管理页面。

验收标准：

- 用户可通过前端完成日常收入、支出、调整和转账记录。
- 转账三种输入模式在前端联动正确。
- 422 业务规则错误能展示可理解提示。
- 不允许前端提交 JSON number 金额。

### 3.4 P2.3 报表与分析增强

目标：

- 增强净资产、现金流、分类统计和 Dashboard Summary。
- 支持周期选择、币种展示、趋势图和导出入口。
- 优化常见数据量下的报表查询性能。

交付物：

- Dashboard Summary 页面。
- 净资产趋势页面。
- 现金流与分类统计页面。
- 报表导出基础能力。

验收标准：

- Transfer 不计入收入/支出统计。
- 缺失汇率时展示“不可用”或明确错误，不参与错误计算。
- 报表查询在常见数据量下响应稳定。

### 3.5 P2.4 数据维护与自愈

目标：

- 完成账户归档、危险删除确认、余额缓存重建和历史汇率补全。
- 提供数据自愈入口，减少人工直接改库。
- 为高风险操作补充审计记录。

交付物：

- 账户归档与恢复流程。
- 危险删除确认流程。
- 余额缓存重建任务。
- 历史汇率补全任务。

验收标准：

- 危险删除必须具备明确确认步骤和审计日志。
- 余额重建结果可复核。
- 数据维护任务失败可重试、可追踪。

### 3.6 P2.5 稳定性、审计与可观测性

目标：

- 建立服务端审计日志、全局异常边界、后台任务监控和告警事件。
- 增强日志结构，支持 TraceId、用户 ID 和任务 ID 关联。
- 为错误路径和后台任务建立追踪入口。

交付物：

- 审计日志查询能力。
- 后台任务状态页面或管理接口。
- 告警事件记录。
- 全局错误边界前后端联动。

验收标准：

- 生产响应不泄露堆栈、SQL、文件路径或密钥。
- 后台任务失败能进入 failed 或 dead letter 状态。
- 管理者可以追踪关键维护动作。

### 3.7 P2.6 质量门禁与发布准备

目标：

- 建立端到端 smoke test、覆盖率门禁、性能基准和文档 lint。
- 梳理发布检查清单和回滚步骤。
- 为 Phase 3 外部接入预留稳定扩展点。

交付物：

- E2E smoke test。
- 覆盖率与性能基准报告。
- 发布检查清单。
- Phase 3 前置能力清单。

验收标准：

- 核心用户路径端到端可跑通。
- 报表、转账、认证和维护任务的关键测试通过。
- 发布前检查项明确，失败时有回滚策略。

## 4. 质量门禁

### 4.1 必跑检查

Phase 2 合并前至少执行：

```text
frontend build
frontend smoke tests
backend build
unit tests
repository integration tests
api tests
e2e smoke tests
markdown check
```

### 4.2 阻断规则

- 前端金额以 JSON number 提交，禁止合并。
- 转账、报表、认证核心路径失败，禁止合并。
- 危险删除、余额重建和审计日志测试失败，禁止合并。
- 全局异常处理泄露敏感信息，禁止合并。

## 5. 风险与应对

### 5.1 前端复制业务规则

风险：前端为了体验便利重复实现复杂金融规则，导致与后端不一致。

应对：

- 前端只做输入约束和交互提示。
- 金融计算结果以后端返回为准。
- 用 API 测试覆盖前后端数据契约。

### 5.2 报表性能退化

风险：数据量增长后，报表查询和折算逻辑变慢。

应对：

- 优先使用数据库端聚合和必要索引。
- 对高频报表增加查询性能基准。
- 缺失汇率时返回明确错误，不做默认折算。

### 5.3 维护入口误操作

风险：危险删除、余额重建等维护能力造成不可恢复的数据问题。

应对：

- 高风险操作必须二次确认。
- 必须写入审计日志。
- 能异步执行的维护任务优先通过后台任务和状态机处理。

## 6. 待决策选项

当前没有必须由维护者立即选择的架构项。

### 6.1 后续可评估事项

- **前端 UI 组件库**：默认沿用设计文档中的 Element Plus；如后续需要更轻量方案，再单独评估。
- **报表导出格式**：优先支持 CSV；是否支持 Excel 和 PDF，可在报表页面稳定后确认。
- **任务管理界面**：可以先提供管理 API，再决定是否做完整后台管理页面。
