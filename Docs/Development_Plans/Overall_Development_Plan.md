# Personal Finance Hub (PFH) - Overall Development Plan

Version: 1.1
Backend: C++23
Architecture: Clean Architecture + Lightweight DDD
Status: Active

---

## 1. 导言与总目标

本文档用于把 PFH 的开发内容划分为三个阶段，先给出总路线和阶段边界。详细阶段计划分别记录在 `Phase_1_Development_Plan.md`、`Phase_2_Development_Plan.md` 和 `Phase_3_Development_Plan.md`。

### 1.1 总体目标

PFH 最终目标是成为一个以个人财务聚合为中心的后端系统：先保证本地记账、账户、转账、汇率和报表的正确性，再增强产品体验与分析能力，最后预留账单导入、支付平台接入和外部同步等扩展能力。

### 1.2 分阶段原则

- **第一阶段先可运行**：完成后端最小闭环，所有核心金融规则必须可测试。
- **第二阶段再好用**：在稳定基础上补齐产品体验、报表能力、审计、运维和前端协同。
- **第三阶段留扩展**：把账单导入、支付平台接入、外部同步等高不确定功能放到预留阶段，避免拖慢核心闭环。

## 2. 三阶段总览

### 2.1 Phase 1：基础闭环与核心正确性

阶段定位：

Phase 1 聚焦“系统能正确记账”。这一阶段优先完成 C++23 后端工程骨架、金融原语、核心领域模型、数据库持久化、事务一致性和最小 REST API。

核心内容：

- 工程骨架：CMake、目录结构、配置、日志和 GoogleTest。
- 金融原语：`Decimal`、`Currency`、`Money`、`ExchangeRate`。
- 领域闭环：用户偏好、账户、流水、转账聚合、余额计算和汇率折算。
- 持久化闭环：PostgreSQL、Flyway、Repository、Unit of Work 和 outbox。
- API 闭环：账户、流水、转账、报表查询和认证基础。
- 质量闭环：单元测试、Repository 集成测试、API smoke test 和 Markdown 检查。

阶段交付：

- 后端可以完成账户创建、流水创建、转账创建、余额查询和基础报表查询。
- 金额与汇率计算不依赖 `float` 或 `double`。
- 关键业务规则具备自动化测试。
- 事务提交、回滚和 outbox 行为可验证。

不纳入本阶段：

- 完整前端产品化体验。
- 外部账单导入。
- 支付平台接入。
- 多节点分布式任务调度。

### 2.2 Phase 2：产品体验、分析增强与稳定性

阶段定位：

Phase 2 聚焦“系统更好用、更可靠”。在 Phase 1 的正确性基础上，完善前端体验、报表分析、审计能力、后台任务、错误处理、运维可观测性和数据维护能力。

核心内容：

- 前端应用：Vue 3、账户管理、流水录入、转账表单、报表页面和用户偏好设置。
- 报表增强：净资产趋势、现金流分析、分类统计、Dashboard Summary 和导出能力。
- 数据维护：账户归档、危险删除确认、余额缓存重建、历史汇率补全和数据自愈入口。
- 体验增强：分类管理、标签管理、搜索过滤、批量编辑和本地化展示。
- 稳定性增强：全局异常边界、审计日志、告警事件、后台任务监控和错误追踪。
- 质量增强：覆盖率门禁、端到端 smoke test、性能基准和文档 lint。

阶段交付：

- 普通用户可以通过前端完成日常记账、查账、转账和看报表。
- 管理和维护动作有明确确认、审计和回滚策略。
- 报表查询在常见数据量下稳定可用。
- 后台任务失败可以被发现、重试和追踪。

不纳入本阶段：

- 真实支付平台写操作。
- 银行、信用卡、支付平台的完整自动同步。
- OCR、PDF、邮件账单解析等高不确定导入能力。

### 2.3 Phase 3：预留扩展与外部生态接入

阶段定位：

Phase 3 是预留功能阶段，用于承接外部数据、自动化同步和平台生态能力。此阶段功能价值高，但不确定性、合规要求和维护成本也更高，因此放在核心系统稳定后再推进。

预留功能：

- 账单导入：CSV、Excel、PDF、邮件账单、银行流水和信用卡账单导入。
- 导入解析：字段映射、金额识别、币种识别、时间解析、商户标准化和重复检测。
- 支付平台接入：支付宝、微信支付、PayPal、Stripe 等平台的账单读取或交易同步。
- 银行与聚合平台接入：开放银行 API、第三方聚合服务和只读账户同步。
- 外部同步框架：Provider 插件化、同步任务调度、幂等指纹、冲突处理和重试队列。
- 自动分类与规则引擎：商户规则、关键词规则、历史行为推荐和批量修正。
- 高级自动化：定期预算提醒、异常交易提醒、订阅识别和现金流预测。

阶段交付：

- 可以通过稳定的导入框架接收外部账单数据。
- 外部数据进入系统前必须经过预览、去重、校验和用户确认。
- 所有外部同步必须保持用户隔离、幂等和可追踪。
- 平台接入失败不影响核心记账能力。

前置条件：

- Phase 1 的核心记账和持久化闭环稳定。
- Phase 2 的用户体验、审计、告警和数据维护能力基本完成。
- 已明确外部数据来源的合规边界和用户授权方式。

---

## 3. 阶段关系

### 3.1 推进顺序

```text
Phase 1: 基础闭环与核心正确性
  -> Phase 2: 产品体验、分析增强与稳定性
    -> Phase 3: 预留扩展与外部生态接入
```

### 3.2 依赖关系

- Phase 2 依赖 Phase 1 的稳定领域模型、Repository、API 和测试基础。
- Phase 3 依赖 Phase 2 的审计、告警、数据维护和用户确认流程。
- 第三方接入不得倒逼 Phase 1 改坏核心金融规则。
